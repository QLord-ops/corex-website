# OpenAI в Project Check (консультация)

В Project Check («Опишите проект») добавлена опциональная интеграция с OpenAI: если в бэкенде задан `OPENAI_API_KEY`, бот **напрямую подключён к OpenAI** — при нажатии «Анализировать» сообщение пользователя обрабатывает GPT и извлекает структурированные поля (цель, тип сайта, сроки, бюджет, интеграции и т.д.). Если ключа нет или запрос к OpenAI завершился с ошибкой, используется прежняя логика на regex.

## Настройка

1. **Ключ загружается из файла** `backend/.env` при старте (python-dotenv). Не нужно экспортировать переменные в shell — подходит для PM2 и локального запуска.

   В `backend/.env` добавьте:
   ```
   OPENAI_API_KEY=sk-...
   ```
2. По желанию укажите модель (по умолчанию `gpt-4o-mini`):
   ```
   OPENAI_EXTRACT_MODEL=gpt-4o-mini
   ```
3. **Свои инструкции для бота** (как в ChatGPT — как должен вести себя ассистент): добавьте в `backend/.env`:
   ```
   OPENAI_PROJECT_CHECK_INSTRUCTIONS=Всегда отвечай кратко. Фокус на малый бизнес. Не предлагай e-commerce, если бюджет низкий.
   ```
   Текст из этой переменной дописывается в системный промпт; модель будет следовать этим правилам при извлечении данных и ответах.
4. **Перезапуск бэкенда** (чтобы подхватить изменения `.env`):
   - Локально: перезапустите процесс (Ctrl+C и снова `uvicorn` или `python -m uvicorn ...`).
   - Под PM2: `pm2 restart corex-backend --update-env` (флаг `--update-env` подхватывает переменные из окружения; если переменные заданы только в `.env`, достаточно перезапуска — сервер при старте читает `backend/.env` через load_dotenv).

## Поведение

- **С ключом:** запрос уходит в OpenAI, ответ (JSON с полями intake) мержится с текущим состоянием; затем считаются `missing_required` и возвращается тот же формат, что и раньше.
- **Без ключа или при ошибке:** используется прежнее извлечение по regex; фронт не меняется.

Фронтенд по-прежнему вызывает `POST /api/extract` с `{ currentIntake, userMessage }` и получает `{ intake, missing_required }`.
